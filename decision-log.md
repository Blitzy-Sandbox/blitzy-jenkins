# Decision Log — Acceleration Report Generation

## Overview

This decision log is the single source of truth for the rationale behind every non-trivial methodological choice made while generating `acceleration-report.md` and its companion executive presentation `acceleration-report.html`. Each row records what was decided, what alternatives were considered, why the chosen alternative was adopted, and what risks that choice carries. The log is mandated by the project-level Explainability rule; it is paired with a bidirectional traceability matrix that maps every Decision ID to the report sections it governs and every major report section to the Decision IDs applied in it.

## Engineer Anonymization Mapping

This section documents the **rule** used to map real author identities to pseudonyms `Engineer A`, `Engineer B`, … for the per-engineer breakdowns in `acceleration-report.md` §6.1 and §6.2. The rule is deterministic and regenerable; the actual author identities are not recorded here, consistent with the user prompt's instruction that the decision log documents the rule, not the identities.

### Regeneration Command

The After-period non-bot non-merge commit ranking that drives the pseudonym assignment is generated by running the following read-only commands against the repository at the same extraction timestamp recorded in `acceleration-report.md` §2:

```bash
# Rank non-bot human authors by After-period non-merge commit volume (descending).
git shortlog -sne --no-merges --since=2025-06-19 \
  | grep -vE '(\[bot\]|jenkins-release-bot|release-bot)' \
  | sort -rn \
  | head -20
```

The top 20 authors from this command form the pool from which Activity 7 §6.2 draws its 10-row breakdown. A comparable command with `--merges --first-parent master` rather than `--no-merges` produces the Activity 3 §6.1 ordering (only 2 engineers produced non-bot merges in the After period, so `Engineer A` and `Engineer B` are the only two pseudonyms assigned for Activity 3). The regeneration command above is exactly `R9.2` in `acceleration-report.md` §10; no additional data source or tool is required.

### Assignment Rule

- The author at rank 1 (highest After-period non-bot non-merge commit volume) is assigned `Engineer A`.
- The author at rank 2 is assigned `Engineer B`.
- Each subsequent rank receives the next letter of the Latin alphabet, in order.
- The mapping is stable across the report and the executive presentation: any row labelled `Engineer A` in §6.1, §6.2, or `acceleration-report.html` Slides 10 and 11 refers to the same underlying author identity.
- The mapping is regenerable at any future time by re-running the command above against the same repository state. A changed ref graph (new commits, removed commits, rebased history) will produce a different mapping; the mapping is therefore a function of the extraction timestamp recorded in §2.

### Author-Category Attribution

The decision log records one factual, non-identifying note about author categories present in the ranking:

- `Engineer A` in the Activity 7 §6.2 ranking has zero Baseline non-merge commits. The associated multiplier is therefore rendered `Insufficient signal — no Baseline commits (division by zero)` per D009. This outcome reflects a first-time contributor whose activity falls entirely within the After period; it is a legitimate artefact of a lifetime-versus-phase comparison and not a data-quality problem.
- Activity 3 §6.1 `Engineer A` and `Engineer B` are maintainers who perform merges on `master`; per §6.1 footnote, this is a maintainer-activity proxy rather than a direct PR-authorship proxy. The pseudonym letters in §6.1 and §6.2 are independent (they rank by different ref sets) and do not necessarily map to the same underlying person.
